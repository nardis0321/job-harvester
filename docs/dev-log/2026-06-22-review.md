### 중복 제거 로직 검토
- 3단계의 중복 판별 기준
  1. `url`
  2. `source + external_id`
  3. `company + title + location`

- 문제 제기 : YANGI
  1. URL이 없는 공고가 수집된다면 비정상 데이터 아닌가
  2. 중복 제거를 위한 과도한 fallback은 문제를 숨길 수 있음. 조기 실패가 버그 발견에 유리.
  3. `company + title + location` 기준은 서로 다른 공고를 잘못 병합할 위험이 있음.

- codex가 url 없는 경우를 고려한 이유
    - 데이터 수집에서 “공고의 고유 식별자”와 “사람이 여는 URL”이 항상 같은 타이밍에 안정적으로 주어지지 않기 때문
    - URL이 없더라도 “어느 사이트의 어떤 공고 ID인지”가 있으면 꽤 강한 중복 제거 키가 된다
    - 다만 이건 external_id가 실제 사이트 내부 ID일 때만 좋은 설계
    - 채용 공고 사이트 탐색 결과, 대부분 공개 페이지 URL에서 ID 패턴이 바로 확인됨

- 결정: 중복 키는
  1. url
  2. source + external_id
  두 가지를 남기고 company + title + location fallback은 제거
